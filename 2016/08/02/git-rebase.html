git-rebase

发表于 2016-08-02   |   分类于 web   |  

注意：一旦分支的快照发布到origin，就千万不要对该分支进行rebase操作。

开发完成后，会有一堆 commit，我们希望合并到主干的时候，只显示一个（或最多两三个）commit，这样不仅清晰，也容易管理。

  1. 首先用git fetch返回服务器上的代码
  2. 首先用git rebase origin/master 合并
  3. 如果发生冲突了会提示，然后可以使用git diff查看冲突，在手工改掉冲突，在用 git add 添加修改后文件，最后用 git rebase –continue 继续没完成的合并
  4. 最后就可以用git push 更新到服务器上去

git rebase 和 git merge 的区别

假设:

  • C3提交于9:00AM;
  • C5提交于10:00AM;
  • C4提交于11:00AM;
  • C6提交于12:00AM;

无论是 rebase 还是 merge 后的commit 都按 时间排序。

对于使用 git rebase 来合并所看到的commit的顺序是：C7 ,C6‘,C5’,C4,C3,C2,C1，因为C6’提交只是C6提交的克隆，C5’提交只是C5提交的克隆，所以 C5’ 生成时间在 C6 之后，C6’ 也在 C5’ 之后。

对于使用 git merge 来合并所看到的commit的顺序是：C7 ,C6,C4,C5,C3,C2,C1。

合并分支

  • 直进式合并（fast forward），不会生成单独的合并节点 git merge hotfix-0.1.1
  • 非直进式合并（none fast-forword），会生成单独节点 git merge --no-ff hotfix-0.1.1

前者不利于保持commit信息的清晰，也不利于以后的回滚，建议总是采用后者（即使用–no-ff参数）。
